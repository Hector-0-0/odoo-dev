================================================================
NOTAS: Carpeta security/ en Odoo
================================================================

La carpeta security/ controla quién puede hacer qué dentro
de tu módulo. Sin ella, los usuarios pueden ver errores de
acceso o directamente no ver los datos aunque el módulo
esté instalado.

No necesita __init__.py porque solo contiene archivos CSV
y XML, no código Python.

Estructura típica:
  mi_modulo/
  └── security/
      ├── ir.model.access.csv   ← permisos CRUD por modelo
      └── security.xml          ← grupos y reglas de acceso

----------------------------------------------------------------
ir.model.access.csv
----------------------------------------------------------------

Es el archivo más importante de la carpeta. Define los permisos
básicos de Create, Read, Update y Delete (CRUD) para cada
modelo del módulo, por grupo de usuarios.

Sin este archivo, cualquier usuario que intente acceder a tus
modelos verá un error "Access Denied" aunque sea administrador.

Estructura de columnas:
  id          → identificador único del registro (nombre interno)
  name        → nombre descriptivo del permiso
  model_id    → referencia al modelo (model_nombre_del_modelo)
  group_id    → grupo que recibe el permiso (vacío = todos)
  perm_read   → 1 permite leer,   0 no permite
  perm_write  → 1 permite editar, 0 no permite
  perm_create → 1 permite crear,  0 no permite
  perm_unlink → 1 permite borrar, 0 no permite

Ejemplo básico (acceso total para todos los usuarios):
  id,name,model_id,group_id,perm_read,perm_write,perm_create,perm_unlink
  access_libro_reclamaciones,access_libro_reclamaciones,model_libro_reclamaciones,,1,1,1,1

Ejemplo con grupo específico (solo administradores):
  id,name,model_id,group_id,perm_read,perm_write,perm_create,perm_unlink
  access_libro_reclamaciones_admin,access_libro_reclamaciones_admin,model_libro_reclamaciones,base.group_system,1,1,1,1
  access_libro_reclamaciones_user,access_libro_reclamaciones_user,model_libro_reclamaciones,base.group_user,1,0,0,0

Reglas para el campo model_id:
  El nombre del modelo usa guiones bajos y el prefijo "model_".
  libro.reclamaciones → model_libro_reclamaciones
  res.partner         → model_res_partner
  sale.order          → model_sale_order

Grupos más usados de Odoo:
  base.group_user        → usuario interno (todos los empleados)
  base.group_system      → administrador técnico
  base.group_portal      → usuario portal (clientes externos)
  base.group_public      → usuario público (sin login)

----------------------------------------------------------------
ir_model_access.xml (formato alternativo al CSV)
----------------------------------------------------------------

Hace exactamente lo mismo que el CSV pero en formato XML.
Algunos desarrolladores prefieren este formato para mantener
todo en XML y no mezclar formatos en la carpeta security/.

Estructura de la etiqueta record:
  <record id="..." model="...">

  id    → identificador externo (XML ID) del registro.
          Nombre único que Odoo usa para identificar este
          permiso. Se usa para referenciarlo con ref="".

  model → tabla de la BD donde se inserta el registro.
          Siempre es "ir.model.access" para permisos CRUD.
          Es equivalente a un INSERT en esa tabla.

Ejemplo básico (acceso total para todos los usuarios):
  <odoo>
      <record id="access_libro_reclamaciones"
              model="ir.model.access">
          <field name="name">access_libro_reclamaciones</field>
          <field name="model_id"
                 ref="model_libro_reclamaciones"/>
          <field name="group_id"
                 ref="base.group_user"/>
          <field name="perm_read"   eval="1"/>
          <field name="perm_write"  eval="1"/>
          <field name="perm_create" eval="1"/>
          <field name="perm_unlink" eval="1"/>
      </record>
  </odoo>

Comparación CSV vs XML (resultado idéntico en la BD):

  CSV:
  id,name,model_id,group_id,perm_read,perm_write,perm_create,perm_unlink
  access_libro_reclamaciones,access_libro_reclamaciones,model_libro_reclamaciones,base.group_user,1,1,1,1

  XML:
  <record id="access_libro_reclamaciones" model="ir.model.access">
      <field name="name">access_libro_reclamaciones</field>
      <field name="model_id" ref="model_libro_reclamaciones"/>
      <field name="group_id" ref="base.group_user"/>
      <field name="perm_read"   eval="1"/>
      <field name="perm_write"  eval="1"/>
      <field name="perm_create" eval="1"/>
      <field name="perm_unlink" eval="1"/>
  </record>

Nota: La etiqueta <record id="..." model="..."> es la forma
      estándar de crear cualquier registro en Odoo desde XML.
      No es exclusiva de permisos, se usa en vistas, menús,
      datos, reportes, etc. Siempre significa lo mismo:
      "inserta o actualiza este registro en esa tabla".

----------------------------------------------------------------
security.xml
----------------------------------------------------------------

Opcional pero recomendado cuando tu módulo necesita diferentes
niveles de acceso. Aquí defines grupos propios y reglas de
dominio para filtrar registros.

Grupos (res.groups):
  Permiten crear roles propios para tu módulo, independientes
  de los grupos base de Odoo.

Ejemplo de grupos propios:
  <odoo>
      <data>
          <!-- Grupo base del módulo -->
          <record id="group_libro_user" model="res.groups">
              <field name="name">Usuario Libro Reclamaciones</field>
              <field name="category_id"
                     ref="base.module_category_hidden"/>
          </record>

          <!-- Grupo administrador que hereda del usuario -->
          <record id="group_libro_manager" model="res.groups">
              <field name="name">Manager Libro Reclamaciones</field>
              <field name="implied_ids"
                     eval="[(4, ref('group_libro_user'))]"/>
          </record>
      </data>
  </odoo>

Reglas de acceso (ir.rule):
  Filtran qué registros puede VER cada usuario, no solo
  si puede acceder al modelo. Usan dominios de Odoo.

Ejemplo (cada usuario solo ve sus propios reclamos):
  <record id="rule_libro_reclamaciones_user" model="ir.rule">
      <field name="name">Libro: ver solo mis reclamos</field>
      <field name="model_id" ref="model_libro_reclamaciones"/>
      <field name="groups" eval="[(4, ref('group_libro_user'))]"/>
      <field name="domain_force">
          [('claim_user_id', '=', user.id)]
      </field>
  </record>

----------------------------------------------------------------
CÓMO SE DECLARA EN EL MANIFEST
----------------------------------------------------------------

Los archivos de security deben declararse en el campo "data"
del __manifest__.py. El CSV de permisos siempre va primero
porque otros archivos pueden depender de los grupos definidos.

Ejemplo:
  "data": [
      "security/ir.model.access.csv",   ← siempre primero
      "security/security.xml",
      "views/reclamacion_views.xml",
      "views/menu.xml",
  ],

----------------------------------------------------------------
ERRORES COMUNES
----------------------------------------------------------------

Error: "You are not allowed to access..."
  Causa:   Falta el modelo en ir.model.access.csv o el
           archivo no está declarado en el manifest.
  Solución: Revisar que el model_id esté bien escrito y
            que el archivo esté en "data" del manifest.

Error: "External ID not found"
  Causa:   El group_id referenciado no existe.
  Solución: Verificar el nombre exacto del grupo en Odoo
            o dejarlo vacío para dar acceso a todos.

Error: columnas mal alineadas en el CSV
  Causa:   El CSV tiene espacios o columnas de más/menos.
  Solución: Verificar que cada línea tenga exactamente
            las 8 columnas separadas por comas.
